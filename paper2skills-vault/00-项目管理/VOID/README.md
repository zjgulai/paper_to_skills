# VOID 框架运转手册

> **核心信条**：第三象限不能被"探索"，只能在已知体系遭遇压力时从裂缝里显现。
> 这个框架不产生答案，只制造**有效摩擦**。

---

## 框架结构

```
VOID = Venture into Original Ignorance Deliberately

V — Venture     主动走到边界（崩溃设计 / 跨行业移植 / 隐性知识访谈）
O — Observe     观察摩擦和断裂（盲点登记 / 放弃问题存档 / 用户异常行为）
I — Interrogate 审讯已有答案（反转逻辑 / 排除人群 / 10年时效测试）
D — Distill     蒸馏出未命名问题（三测试 / 问题分级 / 原型Skill转化）
```

## 运转节奏

```
每季度一次 VOID Session（半天，独立于日常工作）
├── Phase V：2 小时
│   ├── V1 压测：选 3 个当季最成熟 Skill 做崩溃设计
│   ├── V2 移植：1 个跨行业结构扫描
│   └── V3 访谈：来自 blind-spot-registry 的人类隐性知识
├── Phase O：1 小时（审阅当季盲点和放弃问题积累）
├── Phase I：30 分钟（审讯 1 个"最确定的答案"）
└── Phase D：30 分钟（问题蒸馏 + 分级入银行）

每月 30 分钟：problem-bank.md 维护审阅

每季度末：将 1 个 Level 1 问题转化为原型 Skill
```

## 与现有 Skill 萃取系统的接口

```
现有系统（第1/2象限）              VOID 系统（第3象限）
─────────────────────              ─────────────────────
当季最成熟 Skill 列表    ──────→   Phase V1 压测输入
                                   
原型 Skill（验证成立）   ←──────   Phase D 输出
    ↓ 进入标准萃取流程
```

## 原型 Skill 与标准 Skill 的核心区别

| 维度 | 标准 Skill | 原型 Skill |
|------|-----------|-----------|
| 问题状态 | 已被命名和研究 | 未被命名，可能无论文 |
| 代码要求 | 完整可运行 | 无要求（概念验证为主） |
| ROI 估算 | 必须量化 | 描述性估算即可 |
| 论文引用 | 必须 | 可无（这是重点） |
| 验证标准 | 代码通过 + 业务场景 | 最小实验设计 + 失败标准 |
| 生命周期 | 永久在库 | 季度评审，可降级或淘汰 |

## 问题三测试（每个候选问题必须通过）

```
测试1 — 新颖性
在 arXiv / Google Scholar / 中国知网 三库同时搜索
→ 无法找到直接对应的研究 → 通过

测试2 — 真实性
现实世界里已造成损失，只是没人把它识别为"问题"
→ 有真实案例或信号 → 通过

测试3 — 可研究性
在原则上可以被研究，即使当前缺数据或缺方法
→ 能想象出研究路径 → 通过
```

## 问题银行分级

```
Level 1 — 即将成熟（1-2年内条件具备，可先做）
Level 2 — 等待条件（3-5年后才能研究）
Level 3 — 范式级别（需要整个行业认知框架改变）
```
