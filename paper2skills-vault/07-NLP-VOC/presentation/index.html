Section 03 · Why Now

问题不是没有客户反馈，而是反馈无法规模化进入决策

VOC 数据存在于多个渠道，但传统规则 + 人工抽查无法承接「全量、多语言、长文本、跨部门」的现实。

Before · Phase 4

规则系统时代的痛点

  • 多源 VOC 分散在 5 个渠道，部门之间口径不一致
  • 规则只能覆盖显性英文关键词，约 18% 数据「读不懂」
  • 约 6.3 万条样本沉睡在数据库，无标签可用
  • 业务方拿到的是 jsonl 文件，不是结构化洞察
  • 管理层缺少统一的客户声音视图与跨部门指标
After · Phase 5–7

AI + 治理 + BI 后的目标态

  • 5 数据源统一 schema，364,569 条全量可计算
  • LLM 闭集打标 + ABSA + Proxy NPS + 55 画像，覆盖率 97.22%
  • Method C 后处理使 precision 修复至 0.896
  • Superset 8 个看板，业务方 10 秒内拿到 Top-3 负向标签
  • 管理层、7 部门共用同一份口径，证据可溯源
Speaker Notes · 03 为什么做

这一页强调「不是 AI 演示，是业务必要性」。Momcozy 跨境母婴在 5 个渠道沉淀了大量 VOC，过去靠规则只能识别约 82% 的内容，剩下 18% 是多语言、长客服工单、跨主题——它们恰好是最有价值的部分。要解决的真问题是：洞察链路过长，决策没法等。

Section 06 · AI Pipeline

AI 能力分层：5 层流水线，不是一次性调用

每层独立可替换，避免 LLM 「黑箱」，下一阶段 v5.0 字典也只需动 L1。

Speaker Notes · 06 AI 能力

这一页要打消「AI 黑箱」担忧。我们没有把所有任务塞给一个 LLM，而是 5 层分工：规则打底、LLM 做语义、ABSA 抽方面、画像规则结构化、Unified 输出。每一层都可以独立 A/B 替换，下一阶段做 v5.0 字典就是单独动 L1。

Section 09 · Superset Architecture

Superset 平台架构：10 分钟可重建的 BI

配置即代码，REST API 工厂脚本生成，迁移成本最小化。

~7h 开发 · 0 LLM 成本
关键决策 1

普通 VIEW 替代物化视图

查询 < 2s 即可满足 Superset 渲染；省掉 REFRESH 调度复杂度，filter 切片实时生效。

关键决策 2

Chart→Dashboard 双 PUT 协议

先 PUT chart 建立 ownership，再 PUT dashboard 写 layout，避免 Superset 「空占位符」陷阱。

关键决策 3

Filter scope 限定同 dataset

跨 dataset filter 会被 Superset 静默忽略。Overview 与 Dept 两套 filter 完全隔离。

Speaker Notes · 09 Superset 架构

3 件事：自家 Docker 起、配置即代码、~7h 开发 0 LLM 成本。三个工程决策（普通 VIEW、双 PUT、filter 同源）也是 Phase 8 必须延续的纪律。

Section 11 · User Flow

典型用户路径：10 秒拿 Top-3

以产品中心 / 品线为例，业务用户自助过滤，无需算法同事写 SQL。

Step 1

打开 Dashboard 3

业务方直接在浏览器打开「VOC · 产品中心/品线」看板，无需登录额外系统。

默认视图：质量感知 20.2k · 易用性 16.4k · 延迟 12.3k

Step 2

选择「情感极性 = 负向」

点击左侧 native filter「情感极性」下拉，勾选「负向」，点击 Apply。

底层：POST /api/v1/chart/data + filter_state

Step 3

Top-3 完全切换

3–5 秒重渲染后，Top-3 从「质量 / 易用 / 延迟」切到「延迟 / 尺码小 / 使用体验差」。

Playwright 实测：Top-3 完全不同，filter 关联正确

Speaker Notes · 11 用户路径

这一页是「价值落地」的具象化。Phase 7 D4 用 Playwright 实测过这条路径的语义：未过滤 vs 过滤后 Top-3 完全不同，证明 filter 与 dataset 关联正确，不是空壳。每部门的看板都已具备这种切片能力。
