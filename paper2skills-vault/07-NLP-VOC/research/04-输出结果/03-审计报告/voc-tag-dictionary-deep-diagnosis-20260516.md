## 二、标签字段宽表（Excel）深度诊断

### 2.1 结构层诊断

#### 问题 1：列数不一致 + 列序错位

| Sheet | 列数 | 与通用表差异 |
|-------|------|-------------|
| 01_通用标签主表 | 37 | 基准 |
| 02~07 品线 Sheet | 43 | **+6 列**：v3.6_渠道权重_×4 + v3.6_版本 + v3.6_状态 |
| 08_映射关系表 | 30 | 独立结构，无 tag_id 主键 |
| 10_Aspect库 | 9 | 独立结构，无 theme_id 主键 |

**根因**：v4.5 治理时在通用表清理了 v3.6 字段，但品线 Sheet 未同步清理。
**后果**：
- 人工维护时需要在不同 Sheet 间「脑补」列对应关系
- 任何自动化读取脚本都要写两套逻辑（通用 vs 品线）
- tag_dict_loader.py 索性只读 4 列（标签ID/英文/情感/AIPL），浪费了字典中 90% 的字段价值

#### 问题 2：28 个 tag_id 跨 Sheet 重复

```
TAG_P1_001 ~ TAG_P1_005：同时存在于「01_通用标签主表」和「02_吸奶器」
TAG_L1_001 ~ TAG_L1_010：同时存在于「01_通用标签主表」和「07_智能母婴电器」
... 共 28 个重复 tag_id
```

**根因**：历史迭代中「通用标签」与「品线标签」未做命名空间隔离。
**后果**：LLM 闭集打标时，如果 prompt 中同时包含通用 Sheet 和品线 Sheet 的重复 tag_id，模型可能随机选择其中一个，导致标签语义漂移。

**示例**：`TAG_P1_001` 在通用表中可能是「性价比差」（购买决策层），在吸奶器品线中可能是「性价比差（吸奶器专用）」——同一 ID 不同语义。

#### 问题 3：72 个中文标签名重复，跨品线同义异 ID

```
「产品变形」出现 4 次（通用/家居家纺/吸奶器/智能母婴电器）
「充不进电」出现 3 次（通用/吸奶器/母婴综合护理）
「结构设计不合理」出现在 4 个 Sheet
```

**根因**：品线标签从存量业务标签清洗而来，未与通用标签做去重合并。
**后果**：
- 全维度分析时，同一中文语义被拆成多个 tag_id，统计口径碎片化
- 业务方看报表时，「产品变形」在吸奶器看一个数、在家居家纺看另一个数，无法横向对比

### 2.2 数据质量诊断

#### 问题 4：审核状态大面积 NULL

| Sheet | 总行数 | 审核状态=NULL | NULL 比例 |
|-------|--------|--------------|----------|
| 01_通用标签主表 | 267 | 191 | **71.5%** |
| 02_吸奶器 | 82 | 78 | **95.1%** |
| 03_内衣服饰 | 57 | 51 | **89.5%** |
| 04_家居家纺 | 52 | 46 | **88.5%** |
| 05_母婴综合护理 | 70 | 58 | **82.9%** |
| 06_喂养电器 | 53 | 46 | **86.8%** |
| 07_智能母婴电器 | 64 | 43 | **67.2%** |

**根因**：v4.5 治理重点是「cleanup 56 行脏数据 + LLM 补齐 230 处空值」，但审核状态字段未被纳入治理范围。
**后果**：「已通过」「已审核-自动填充」的标签只有 109 个（16.9%），其余 536 个标签的审核状态未知。自动打标时无法区分「已验证可用」vs「待审核」vs「已废弃」。

#### 问题 5：品线 Sheet 审计字段 100% NULL

```
合理性评分: 100% NULL（全部 6 个品线 Sheet）
风险等级: 100% NULL
问题诊断: 100% NULL
优化建议: 100% NULL
优化优先级: 100% NULL
```

**根因**：这些字段在 Phase 5-6 的评估流程中产出（通过 5K 抽样 + LLM 评估），但只回填到了通用标签主表，品线标签未执行同样评估。
**后果**：品线标签（378 个，占总数 58.6%）处于「零审计」状态，无法判断哪些品线标签存在覆盖过高、关键词混淆、置信度不稳定等问题。

### 2.3 语义层诊断

#### 问题 6：映射关系表覆盖率 62.8%，203 个标签「失联」

```
字典中唯一中文标签总数: 537
映射表中唯一中文标签数: 337
覆盖率: 62.8%
未映射标签数: 203（37.2%）
```

**未映射标签示例**：湿巾加热盒、吸奶器测评关注、推荐给其他妈妈、可转换设计、温奶器温度准确、性能差、App时间记录错误、吸鼻器清洁方便、护理品牌信任、功能失效...

**根因**：映射关系表（08 Sheet）是人工维护的业务映射，新增标签时未强制要求同步更新映射表。
**后果**：
- 这 203 个标签无法归因到「主责部门」「策略包」「故事线关联」
- BI 看板中这些标签的统计结果无法转化为业务动作
- 自动打标后，全维度分析缺少「部门维度」「策略维度」的切片能力

#### 问题 7：Aspect 库仅 55 个 Aspect，Tag→Aspect 覆盖率 ~26%

```
Aspect 总数: 55
Aspect→Tag 总关联数: 167
平均每个 Aspect 关联 Tag 数: 3.0
645 个 Tag 中，有 Aspect 关联的估计比例: ~25.9%
```

**根因**：Aspect 库在 Phase 5 D4 作为 ABSA 三元组抽取的辅助产物建立，但未随字典迭代同步扩展。
**后果**：
- 74.1% 的 Tag 没有 Aspect 归属，全维度分析时缺少「方面/维度」这一核心聚合层
- 业务方想做「客服方面有哪些问题」「物流方面用户怎么说」时，只能人工翻标签，无法自动化聚合
- ABSA 抽取的 (aspect, sentiment, confidence) 三元组与标签字典的映射关系未建立

## 三、项目架构与流程诊断

### 3.1 字典→LLM 管道的适配性

**现状**：`tag_dict_loader.py` 将 645 个标签压缩为 4 字段（TAG_ID|英文|情感|AIPL），约 ~6,000 tokens 的 system prompt。

```python
# 当前 compact prompt 格式
TAG_L1_001|disconnects within 3 meters-|L1
TAG_L1_002|app auto-runs without request-|L1
...
```

**问题**：
1. **字段严重裁剪**：标签定义、关键词、适用品线、适用载体、用户画像等 30+ 字段全部丢弃，LLM 只能靠「英文缩写」猜测标签语义
2. **路径指向 v3.9**：`DICT_PATH = .../tag_dictionary_v3.9.xlsx`，而当前生产字典是 v4.5，存在版本漂移风险
3. **无 aspect/theme 层次**：LLM 一次性面对 645 个平铺标签，缺少「先定位 Aspect → 再定位 Theme → 最后选 Tag」的分层决策路径

### 3.2 字典→BI 分析的适配性

**现状**：Superset BI 通过 6 个 SQL 视图聚合打标结果，视图基于 `voc_label` 和 `dim_tag` 表。

**问题**：
1. **`dim_tag` 表缺少 Aspect 和 Theme 字段**：当前只有 tag_id / 标签中文 / 标签英文 / 情感极性 / AIPL 节点 / 主责部门 / 策略包，缺少「标签主题」和「Aspect」两列
2. **主题（标签主题）在字典中存在但未进 BI**：字典中有「产品核心性能」「使用舒适度」等主题，但 BI 视图未暴露，导致无法按主题聚合
3. **品线维度（product_line）2026-05-14 刚加入 dim_tag**：已有数据，但 Superset charts 尚未利用此维度做交叉分析

### 3.3 自动打标→全维度分析的链路缺口

```
当前链路：
评论文本 → LLM 打标（645 Tag 闭集）→ Method C 过滤 → BI 视图 → Dashboard
                    ↑
              缺失：Aspect 层聚合
              缺失：Theme 层聚合
              缺失：Tag→业务动作自动映射

目标链路：
评论文本 → LLM 分层打标（Aspect → Theme → Tag）→ Method C 过滤
              → 结构化宽表（review × aspect × theme × tag × sentiment × dept × strategy）
              → 全维度分析（部门/策略/主题/Aspect/品线/画像/渠道/时间）
```

## 四、核心问题优化方案

### 方案 A：字典治理层（数据层）

#### A1. 统一 Schema：所有 Sheet 合并为 1 张「主表」+ 3 张「辅表」

**推荐结构**：

```
tag_dictionary_v5.0.xlsx
├── 00_字段说明
├── 01_TAG主表          ← 645 行 × 20 列（所有标签统一一张表）
├── 02_品线映射表        ←  tag_id × product_line（多对多）
├── 03_Aspect映射表      ←  aspect_id × tag_id（多对多）
├── 04_Theme映射表       ←  theme_id × tag_id（多对多）
└── 05_存量标签归档
```

**01_TAG主表 20 列设计**：

| # | 字段名 | 类型 | 说明 |
|---|--------|------|------|
| 1 | tag_id | PK | 全局唯一，命名空间隔离（通用前缀 GEN_，品线前缀用产品线码） |
| 2 | tag_name_cn | 必填 | 中文标签名 |
| 3 | tag_name_en | 必填 | 英文标签名 |
| 4 | aipl_node | 必填 | A/I/P1/P2/L1/L2/L3 |
| 5 | theme_id | FK | 关联 Theme 表 |
| 6 | sentiment | 必填 | 正向/负向/中性 |
| 7 | ai_extractable | 必填 | 是/否 |
| 8 | source_type | 必填 | 新建/存量清洗/自动发现 |
| 9 | keywords_en | 必填 | LLM 匹配用英文关键词 |
| 10 | keywords_cn | 必填 | 规则匹配用中文关键词 |
| 11 | tag_definition | 必填 | 标签定义（LLM 理解用） |
| 12 | applicable_product_lines | 多值 | 逗号分隔品线名 |
| 13 | applicable_channels | 多值 | 逗号分隔渠道 |
| 14 | persona_profile | 多值 | 适用用户画像 |
| 15 | atomic_indicator_id | 可空 | 对应原子指标 |
| 16 | metric_direction | 可空 | positive/negative |
| 17 | proxy_nps | 可空 | promoter/detractor/passive |
| 18 | is_generic | 必填 | 是=通用 / 否=品线专属 |
| 19 | audit_status | 必填 | 已通过/待审核/已废弃/自动填充 |
| 20 | dept_owner | 可空 | 主责部门 |

**好处**：
- 消除 Sheet 间列数/列序差异
- 消除 tag_id 跨 Sheet 重复（统一命名空间）
- tag_dict_loader.py 只需读 1 张 Sheet
- 新增标签时只需在 1 处维护

#### A2. 命名空间隔离：解决 tag_id 重复

**规则**：
- 通用标签：`TAG_GEN_[AIPL]_[序号]`（例：`TAG_GEN_L1_001`）
- 品线标签：`TAG_[品线码]_[AIPL]_[序号]`（例：`TAG_BREAST_L1_001` 吸奶器）
- 品线码：BREAST（吸奶器）/ UNDER（内衣）/ HOME（家居）/ CARE（护理）/ FEED（喂养）/ SMART（智能）

**迁移脚本**：自动扫描重复 tag_id，按所属 Sheet 重命名，同步更新：
- 所有 jsonl 打标结果中的历史 tag_id
- BI 数据库 dim_tag 表
- Superset dataset 定义

#### A3. 强制映射回填：解决 203 个标签「失联」

**流程**：
1. 用 LLM 自动推断未映射标签的「主责部门」「策略包」「故事线关联」
2. 人工审核 LLM 推断结果（预计 2 小时）
3. 审核通过后写入映射表
4. 建立「新增标签必须同步更新映射表」的 CI 门禁

#### A4. Aspect 库扩展：从 55 扩展到 150+

**方法**：
1. 基于 645 个标签的「标签主题」字段做聚类（现有主题约 30 个）
2. 将每个「标签主题」提升为 Aspect（如「产品核心性能」→ Aspect「产品性能」）
3. 用 LLM 辅助审查每个 Tag 与 Aspect 的归属关系
4. 目标：Aspect 数 55 → 120，Tag→Aspect 覆盖率 26% → 95%

### 方案 B：工程架构层（代码层）

#### B1. tag_dict_loader.py 升级：支持分层提示

**当前**：平铺 645 个标签 → LLM
**目标**：分层提示（3 层）

```
Layer 1: Aspect 层（~120 个 Aspect）
  "请判断这段评论涉及以下哪些方面：客服/物流/产品性能/价格/..."

Layer 2: Theme 层（每个 Aspect 下 ~5 个 Theme）
  "在「产品性能」方面，涉及以下哪些主题：核心功能/耐用性/精度/..."

Layer 3: Tag 层（每个 Theme 下 ~5-10 个 Tag）
  "在「核心功能」主题下，涉及以下哪些标签：不出雾/不搅拌/不震动/..."
```

**预期收益**：
- LLM 决策路径从「645 选 N」变成「120 选 M → 30 选 K → 100 选 N」，准确率提升
- 分层结构天然支持「全维度分析」的聚合维度
- Prompt 长度减少 ~40%（每层只需加载当前分支的标签）

#### B2. 结构化宽表：review × dimension 全维度表

**新建表**：`voc_review_dimensions`

| 字段 | 说明 |
|------|------|
| review_id | 原评论 ID |
| source | Amazon/Trustpilot/Zendesk/Momcozy/Reddit |
| product_line | 品线 |
| aspect_id | Aspect 维度 |
| theme_id | Theme 维度 |
| tag_id | Tag 维度 |
| sentiment | 情感 |
| confidence | 置信度 |
| dept_owner | 主责部门 |
| strategy_package | 策略包 |
| story_line | 故事线 |
| persona_tags | 画像标签（JSONB）|
| proxy_nps | Proxy NPS |

**好处**：
- 一行评论可拆多行（多标签 → 多维度记录）
- Superset 可直接对此表做任意维度交叉分析
- 无需在 BI 层做复杂 JSON 解析

#### B3. 字典版本控制：Excel → YAML/JSON

**问题**：Excel 无法版本控制、diff、CI 校验
**方案**：

```
tag_dictionary_v5.0.yaml   ← 主字典（Git 版本控制）
tag_dictionary_v5.0.xlsx   ← 导出视图（供业务方阅读）
```

**YAML 结构**：

```yaml
tags:
  - tag_id: TAG_GEN_L1_001
    name_cn: 品牌陌生
    name_en: brand feels unfamiliar
    aipl: L1
    theme_id: THEME_001
    aspect_ids: [ASP_001, ASP_002]
    sentiment: negative
    keywords_en: ["brand unknown", "never heard of"]
    keywords_cn: ["没听过", "不认识这个品牌"]
    definition: "用户在第一次接触品牌时表达陌生和认知不足"
    product_lines: [通用]
    channels: [社媒, 广告评论, 品牌词搜索]
    audit_status: approved
    dept_owner: 品牌市场中心
    strategy_package: 品牌认知校准包

themes:
  - theme_id: THEME_001
    name: 品牌初始印象
    aspect_id: ASP_001

aspects:
  - aspect_id: ASP_001
    name_cn: 品牌认知
    name_en: brand awareness
```

**CI 校验**：
- tag_id 唯一性检查
- theme_id/aspect_id 外键检查
- 必填字段完整性检查
- 中英文关键词非空检查
- aipl_node 枚举值检查

### 方案 C：分析洞察层（BI 层）

#### C1. Superset 新增 3 个核心 Dashboard

| Dashboard | 维度 | 用途 |
|-----------|------|------|
| Aspect 全景分析 | Aspect × 情感 × 时间 × 品线 | 高层业务洞察，回答「用户最关心哪些方面」 |
| Theme 深度分析 | Theme × 标签 × 渠道 × 画像 | 中层运营洞察，回答「每个主题下哪些具体问题最突出」 |
| 部门行动看板 | 部门 × 策略包 × 故事线 × 优先级 | 底层执行洞察，直接输出「哪个部门该做什么」 |

#### C2. 自动化周报升级：从 7 部门 → Aspect+Theme 双轴

**当前**：7 部门 × AGRS + MAA（28 文件）
**目标**：7 部门 + 8 Aspect + 15 Theme 三轴交叉

**新增产出**：
- Aspect 周报：每个 Aspect 的趋势、情感分布、Top Tag、新增问题
- Theme 周报：每个 Theme 的归因分析、跨品线对比、策略建议
- 部门-Theme 矩阵：哪个部门在哪些 Theme 上问题最多
